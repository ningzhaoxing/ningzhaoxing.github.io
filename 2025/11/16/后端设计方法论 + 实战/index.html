后端系统设计方法论 + 实战

发表于 2025-11-16 分类于 系统设计 本文字数： 1.8k 阅读时长 ≈ 7 分钟

在面试以及项目实战中，能够根据需求设计出一套可用的、可靠的、高性能、安全、好扩展的系统/模块是后端设计能力的基本功。
下面通过一点方法论和实战场景，来帮助我们更流畅地设计出一个技术方案。

方法论

1. 需求拆解

1.1 功能性需求

必须满足实现的功能。

1.2 非功能性需求

性能、可靠性、安全性、扩展性等。

  1. 异常处理（网络超时、依赖服务失败等）：进行重试机制、降级策略
  2. 安全性：token 校验、权限验证、数据加密（用户密码等）
  3. 性能优化：高频查询接口加缓存、数据库层面加索引、非核心流程上消息队列异步执行
  4. 扩展性：数据模型设计预留字段

2. 系统架构设计

2.1 模块划分

  • 模块拆分。每个模块单一职责（负责独立的业务模块）、降低耦合
  • 明确模块依赖关系。比如订单模块依赖商品模块查询价格；依赖支付模块确认支付状态等

2.2 架构选择

  • 小型系统：单体架构
  • 中大型系统：微服务架构。按需选择是否将该模块拆分为微服务，这样可以提供更大的灵活性，比如不同服务的性能运行合适的微服务；提高服务可靠性，即便一个服务挂了，也不会影响其他服务；提高性能，支持通过异步通信的方式提高吞吐量。但同时也需要考虑微服务带来的系统开销、分布式事务、服务注册发现、负载均衡等问题。
  • 中间件选择：缓存（redis）、消息队列（rocketmq/kafka）、数据库（主从/分库分表）、搜索引擎（Es）

3. 数据实体设计

  • 实体的生命周期管理。数据实体设计支撑业务流程，包含实体的生命周期、缘起缘灭。
  • 实体的存储与查询。确保数据的存储、高效查询
  1. 核心实体的定义。从业务流程中抽取出核心实体，比如订单、商品、用户等。
  2. 实体关系设计。用户与订单是一对多关系、商品与订单是多对多关系等。
  3. 数据库表设计。明确数据表的主键、外键（与哪些实体关联）、索引、字段类型（varchar、金额用 decimal 而非 float、timestamp 等）

4. 接口设计

接口是不同服务之间交互的桥梁（常见的是前后端交互、当然也有微服务之间的交互（RESTful风格的 HTTP、 RPC 等））。

  1. 接口设计原则。
    • 遵循 RESTful 风格。
    • 幂等性。重复调用接口结果一致，比如支付接口需要防止重复支付，可以通过订单 ID 来实现。
  2. 接口的入参和出参
  3. 后端逻辑处理
    • 前端参数校验
    • 业务逻辑校验
    • 业务逻辑处理

5. 补充考量

补充考量的内容，一般为非功能性的需求。用于保证系统的可靠性、安全性、扩展性以及提高系统性能等。

  1. 异常处理。需要有重试机制、降级策略，来处理网络拥塞超时、依赖服务不可用等问题。
  2. 安全性：敏感接口需进行认证（token）、授权（权限管理）等
  3. 扩展性：数据实体设计预留字段；通过设计模式、预留接口为可能频繁变动的实现做预留准备。
  4. 性能优化：高频读写加缓存；异步处理非核心流程等

场景实战

1. 设计实现一个文章审核机制

需求：

已知数据实体：

  • User(id, name)
  • Post(id, author_id, content, visibility, created_at). visibility 为枚举值：PUBLIC/PRIVITE
  • Admin(id, name)
  • Friendship(usera_id, userb_id)
    需求：
    文章审核机制。用户发布 visibility 为 PUBLIC 的文章需要被管理员审核通过后才能被所有人查看，否则只能被用户自己或用户好友（信息流形式）看到。

参考答案/分析过程

1. 需求拆分

  • 功能性需求：
    • 用户发布为 PUBLIC 的文章需要被审核
    • 管理员可以审核 PUBLIC 的文章，审核通过/拒绝
    • 审核中/未审核通过的 PUBLIC 文章只能被用户自己和好友看到
  • 非功能性需求
    • 审核结果异步通知，不影响发布主流程
    • 信息流场景，根据情况是否需要提高查询性能

2. 系统架构设计

  • 可以根据实际接口响应时间情况，决定是否引入缓存（redis），加速查询

3. 数据实体设计

  1. 实体定义：实体 Post 新增字段 review_status 表示审核状态，枚举值：PENDING、APPROVED、REJECTED。
  2. 生命周期：
    • 当用户创建一个 PUBLIC 属性的文章时，review_status 为 PENDING
    • 管理员和作者、作者好友可以查看 PENDING 属性的文章
    • 管理员可以修改 PENDING 属性文章的状态为 APPROVED/REJECTED
  3. 数据表设计：
    • Post 表新增字段：
      • review_status：表示文章的审核状态
      • reviewer_id：表示审核人，关联 admin 表的外键
      • review_at：审核时间
    • Post 表新增索引：
      • 新增 (author_id ,visibility ,review_status) 三个字段的联合索引（用于作者、管理员和作者好友的查询）

4. 接口设计

  1. 修改原发布文章接口请求体（POST /posts）

请求体：

{
	"content": "",
	"visibility": "PUBLIC"
}

后端逻辑：

  • 如果 visibility 为 PUBLIC 则需要将 review_status 设置为 PENDING
  • 如果为 PRIVIATE ，则直接设置为 APPROVED

响应体：

{
	"success": true,
	"message": "发布成功，待审核",
	"data": {
		"new_review_status": "PENDING"
	}
}
  1. 新增管理员查询接口（GET /admin/posts/pending?page=1&size=20）

后端逻辑：

  • 权限校验
  • 分页查询待审核文章

响应体：

{
  "success": true,
  "message": "查询成功",
  "data": {
	  "total": 50,
	  "posts": [
	    {
	      "post_id": 1001,
	      "author_id": 2001,
	      "author_name": "张三",  // 关联用户表查询
	      "content": "今天天气很好",
	      "created_at": "2025-11-16 10:00:00"
	    }
	  ]
  }
}
  1. 管理员审核接口（PUT /admin/posts/:post_id/review）

请求体:

{
	"review_status": "APPROVED",
	"reviewer_id": 1
}

后端逻辑：

  • 前端参数校验
  • 文章是否存在
  • 检查该文章审核状态是否为 PENDING 状态
  • 更新 review_status、reviewer_id、reviewed_at字段
  • 结果异步通知给用户

响应体：

{
	"success": true,
	"message": "审核通过",
	"data": {
		"post_id": 1,
		"new_review_status": "APPROVED",
		"reviewer_id": 1,
		"reviewed_at": "xxxx"
	}
}
  1. 查看作者文章（GET /users/:user_id/posts?page=1&page_size=20）

后端逻辑：

  • 查询作者所有文章
  • 判断用户身份
    • 用户为作者：全部可见
    • 用户为作者好友：可见 PUBLIC 的所有文章
    • 用户为非作者好友：可见 PUBLIC 且 APPROVED 的文章
{
  "success": true,
  "message": "查询成功",
  "data": {
	  "total": 20,
	  "posts": [
	    {
	      "post_id": 1001,
	      "author_id": 2001,
	      "author_name": "张三",  // 关联用户表查询
	      "content": "今天天气很好",
	      "review_status": "PENDING",
	      "created_at": "2025-11-16 10:00:00"
	    }
	  ]
  }
}

5. 补充考量

  1. 性能优化：
    • 给 (author_id ,visibility ,review_status) 三个字段添加联合索引，避免全表扫描
    • 如果查询文章接口有响应时间瓶颈，可以增加缓存
  2. 用户体验：
    • 审核完成后，将结果可以异步通知给用户（发布订阅模式等）
  3. 扩展性：
    • review_status 枚举类型，后续如果需要二次审核等，可以低成本动态扩展
